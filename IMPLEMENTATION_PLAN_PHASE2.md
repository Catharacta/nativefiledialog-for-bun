# 実装計画: フェーズ2 - FFI (Bun.dlopen) によるネイティブバインド

このドキュメントでは、`nativefiledialog-extended` の共有ライブラリを Bun の FFI 機能を用いて直接呼び出すための実装計画について詳述します。

## 1. 目的
現在のスクリプトベース（PowerShell/osascript/zenity）の実装を、共有ライブラリ（.dll/.dylib/.so）を直接呼び出す FFI 実装にアップグレードします。これにより、以下のメリットが得られます：
- パフォーマンスの向上（プロセス起動オーバーヘッドの解消）
- より詳細な制御（デフォルトファイル名の指定など、スクリプト版では制限のある機能の利用）
- 外部コマンドへの依存排除（ライブラリ単体での動作）

## 2. 技術スタック
- **Runtime:** Bun (FFI 機能: `Bun.dlopen`)
- **Library:** [nativefiledialog-extended](https://github.com/btzy/nativefiledialog-extended) (C API)

## 3. 主要な設計項目

### A. 共有ライブラリのロードロジック
- 各プラットフォームに対応したライブラリ名とパスを解決します。
    - Windows: `nfd.dll`
    - macOS: `libnfd.dylib`
    - Linux: `libnfd.so`
- パッケージ内の `bin/` ディレクトリ配下に配置されたバイナリを優先的にロードし、失敗した場合はシステムパスを検索します。

### B. C API と TypeScript のマッピング
`nfd.h` の関数を `Bun.dlopen` で以下のように定義します：

| C 関数名 | Bun FFI 型定義 (args / returns) | 説明 |
| :--- | :--- | :--- |
| `NFD_Init` | `[]`, `i32` | ライブラリの初期化 |
| `NFD_Quit` | `[]`, `void` | ライブラリの終了処理 |
| `NFD_OpenDialog` | `["ptr", "ptr", "u32", "ptr"]`, `i32` | ファイル選択ダイアログ |
| `NFD_PickFolder` | `["ptr", "ptr"]`, `i32` | フォルダ選択ダイアログ |
| `NFD_FreePath` | `["ptr"]`, `void` | 取得したパス文字列のメモリ解放 |

### C. データ構造の変換
- **フィルタ項目の変換:** `nfdfilteritem_t` 構造体の配列を Bun の `Buffer` または `Uint8Array` を用いてメモリ上に構築し、ポインタとして渡します。
- **文字列変換:** TypeScript の文字列（UTF-16）と C API の文字列（UTF-8）の相互変換を行います。

### D. メモリ管理
- C 側で確保されたメモリ（`outPath` など）は、取得後に必ず `NFD_FreePath` を呼び出して解放します。
- 複数選択 (`NFD_PathSet`) の場合は、ループでパスを取得した後に `NFD_PathSet_Free` を実行します。

## 4. 実装手順
1.  **バイナリの準備:** 各プラットフォーム向けの共有ライブラリを `bin/` フォルダに配置します。
2.  **FFI バックエンドの実装:** `src/backends/ffi.ts` を新規作成し、`Bun.dlopen` によるバインディングを記述します。
3.  **メインエントリポイントの更新:** `src/index.ts` の `getBackend()` を修正し、FFI版を優先的に使用するようにします。
4.  **フォールバックの実装:** FFI版のロードに失敗した場合、フェーズ1で作成したスクリプト版を自動的に使用する仕組みを導入します。

## 5. 検証プラン
- **単体テスト:** FFI 経由での関数呼び出しがセグメンテーションフォールトを起こさず正しく動作することを検証。
- **リソースリークテスト:** 大量のダイアログ呼び出しとキャンセルを繰り返し、メモリ使用量が増大しないことを確認。
